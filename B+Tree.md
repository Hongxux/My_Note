**虽然二叉树和红黑树是经典的数据结构，但由于其“深度”问题在磁盘I/O场景下会成为性能瓶颈，因此不适合直接用作数据库的索引结构。​**​ 而B+Tree通过其“多路、矮胖”的特性，完美地解决了这个问题，成为了数据库索引的实际标准。

| 核心挑战          | 二叉树/平衡二叉树（如AVL、红黑树）的不足                                                   | B+树的解决方案                                                                                                                                      |
| ------------- | ------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------- |
| **磁盘I/O效率**​  | 树的高度较高，**磁盘I/O次数近似为O(log₂n)**。每次访问一个节点都可能需要一次磁盘I/O，当数据量大时，随机I/O次数多，效率低下。 | **降低树高**：通过**多路平衡树**结构，一个节点可包含大量键值，极大降低树的高度，从而**显著减少I/O次数**。                                                                                  |
| **利用磁盘预读**​   | 节点大小通常不匹配磁盘块，**无法有效利用磁盘顺序读的高效性**。每次读取可能只获取少量有用数据。                        | **节点大小匹配磁盘页**：将节点大小设置为磁盘页（如4KB）的整数倍。一次I/O可加载多个键，**充分利用预读特性，减少I/O次数**。                                                                         |
| **范围查询效率**​   | 进行范围查询（如查找某个范围内的所有数据）时，效率较低，可能需要在树中进行多次中序遍历，导致大量随机I/O。                   | **叶子节点顺序链表**：所有叶子节点通过**指针串联成有序链表**。范围查询只需在链表上顺序遍历，**大量顺序I/O，效率极高**。                                                                           |
| **稳定性与存储效率**​ | 数据可能存储在所有节点中，树的结构会因频繁更新而不断调整。                                            | **稳定的查询性能**：所有数据记录只存储在**叶子节点**，任何查询从根到叶子的路径长度相同，**查询性能稳定**。<br>**更高的扇出**：非叶子节点仅存储键（索引信息）和指向子节点的指针，不存储实际数据，使得单个节点能**容纳更多的键（更高的“扇出”）**，树高进一步降低。 |
- **应对大规模数据**：对于海量数据（如十亿级别），B+树通过3-4次I/O即可定位到数据，而二叉树可能需要几十次，性能差异巨大
- **适合数据库操作模式**：数据库中存在大量范围查询（如`BETWEEN`、`>`、`<`等操作），B+树的顺序链表结构为此提供了原生支持

----
 - 节点的存储
	 - 存储特点：B+Tree节点大小设置为与磁盘物理页（如16KB）对齐
	 - 服务场景：进行范围查询
		 - 读取某个B+Tree节点（即一个页），它向磁盘发起一次I/O请求，磁盘预测性地将紧随其后的若干连续页（比如接下来的48KB或64KB数据）也一并加载到数据库的内存缓冲池中，避免了后续的磁盘I/O。
	 - 优化原理：利用[[磁盘预读特性]]
 - 节点类型与功能
	- ​**非叶子节点（图示上方的蓝色矩形）​**​：
		- 职责：引导查询快速定位到包含目标数据的叶子节点
		- 存储特点：不直接存储数据行的完整信息，而是存储索引键值和指向下一层子节点的指针
			- 核心设计目的：减少磁盘I/O次数
			- 实现方式：提高扇出，从而降低树的高度
				- 更大的“扇出”：在磁盘一次读取的数据量（通常为一页，例如16KB）固定的情况下，一个B+树的非叶子节点因为不存储数据，因此可以容纳**更多的键值和指针**
				- 从而降低树的高度：扇出越大，树的层级（高度）就越低
			- 次要设计目的：保证查询效率的稳定性
			- 实现原理：所有查询都必须从根节点遍历到叶子节点才能拿到数据，从而任何一条数据的查询路径长度都是相同的
	- ​**叶子节点（图示最下方的绿色矩形）​**​：
		- 职责：存储数据或者指向数据的指针
		- 结构特点：所有叶子节点之间通过**指针双向连接**，形成一个有序链表
			- 更适合范围查询: 叶子节点之间通过双向链表连接，使得进行范围查询时非常高效
				- 工作模式：找到起始点后，只需沿着链表遍历即可，无需回溯树的上层结构。
				- 对比：B树做范围查询则需要复杂的中序遍历。
			- 更方便的全表扫描: 直接遍历叶子节点的链表即可完成全表扫描，比B树效率高。
- 组织形式：有序性
	- 含义：无论是内部节点还是叶子节点，其内部的键值都是有序排列的 
	- 作用：这为范围查询和排序操作提供了天然的支持。
