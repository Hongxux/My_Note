 #### 1. ​**CHAR 与 VARCHAR：定长与变长的根本区别**​

- ​**CHAR(N)​**​：
    
    - ​**定长**​：无论实际内容多少，始终占用 N 个字符的存储空间（不足部分用空格填充）。
        
    - ​**特点**​：读取速度快，但可能浪费存储空间。
        
    - ​**适用场景**​：长度基本固定的字段，如 MD5 哈希值（32位）、国家代码（如 'CN', 'US'）、枚举代码（如 'M', 'F'）。
        
    
- ​**VARCHAR(N)​**​：
    
    - ​**变长**​：占用空间为「实际数据长度 + 长度记录开销（1或2字节）」。
        
    - ​**特点**​：节省存储空间，但读取和更新性能略低于 CHAR。
        
    - ​**适用场景**​：长度变化较大的字段，如用户名、电子邮件地址、文章标题。
        
    

​**工程实践建议**​：在定义 VARCHAR 长度时，应基于业务需求设置合理的最大值，而非一味使用很大值（如直接 65535），因为这会影响内存临时表等组件的使用效率。

#### 2. ​**TEXT 与 BLOB：文本与二进制的存储**​

- ​**TEXT 系列（TINYTEXT, TEXT, MEDIUMTEXT, LONGTEXT）​**​：
    
    - 存储纯文本字符串，受数据库字符集（如 utf8mb4）和排序规则影响。
        
    - 适合存储文章内容、日志、JSON 字符串等。
        
    
- ​**BLOB 系列（TINYBLOB, BLOB, MEDIUMBLOB, LONGBLOB）​**​：
    
    - 存储二进制数据，无字符集概念，字节级原始存储。
        
    - 适合存储图片、PDF 文件、序列化对象等二进制内容。
        
    

​**重要注意事项**​：对于 TEXT/BLOB 类型，通常建议将这类大字段单独存放到一张扩展表中，并与主表通过外键关联。这可以避免大字段拖慢主表的查询速度（因为数据库以页为单位加载数据，大字段会减少单页内存储的主记录条数）。

#### 3. ​**存储容量层级**​

字符串类型按容量分为四个清晰的层级，便于根据数据大小进行选择：

1. ​**微型级 (TINYXXX)​**​：255 Bytes。适用于极短的字符串或微型二进制数据。
    
2. ​**常规级 (XXX)​**​：64 KB。满足大多数长文本或二进制对象的存储需求。
    
3. ​**中型级 (MEDIUMXXX)​**​：16 MB。适用于较大的文档、高清图片或详细的文本内容。
    
4. ​**巨型级 (LONGXXX)​**​：4 GB。用于存储极大的文件，如视频、大型备份数据等。
    

### 经典使用场景与选择指南

|场景描述|推荐类型|理由|
|---|---|---|
|存储固定长度的标识码（如身份证号）|CHAR(18)|长度固定，使用 CHAR 无存储浪费且读写快。|
|存储用户昵称|VARCHAR(50)|长度可变，VARCHAR 更节省空间。|
|存储用户头像图片文件|MEDIUMBLOB|头像大小通常在几MB内，MEDIUMBLOB 的 16MB 容量足够。|
|存储新闻文章内容|MEDIUMTEXT|文章内容可能很长，但一般不会超过 16MB。|
|存储产品手册 PDF 文件|LONGBLOB|产品手册可能很大，需要支持 4GB 的存储容量。|
