
|对比维度|@Scheduled (Spring 内置)|Quartz (企业级框架)|
|---|---|---|
|**定位与复杂度**​|**轻量级**，与 Spring 无缝集成，开发简单|**功能强大**，支持复杂的任务调度和管理，配置相对复杂|
|**分布式支持**​|**单机模式**：多实例部署会导致任务**重复执行**​|**集群模式**：通过数据库锁实现任务**互斥执行**，避免重复|
|**任务持久化**​|**不支持**。任务信息存于内存，应用重启后任务和状态会丢失|**支持**。可将任务信息持久化到数据库，应用重启后**可恢复**​|
|**动态调整**​|**不支持**。修改执行时间通常需要重启应用|**支持**。可通过 API **动态**添加、修改、暂停和恢复任务|
|**任务依赖与高级特性**​|**不支持**复杂的依赖管理、分片等|**支持**任务链、日历调度、故障转移、负载均衡等**企业级特性**​|
|**适用场景**​|简单的、单机的、无需持久化的定时任务|复杂的、分布式的、需要高可靠性和持久化的企业级任务调度|
### 💡 如何选择？

面对这两种方案，你可以遵循一个简单的原则：

- **选择 `@Scheduled`**：如果你的应用是**单实例部署**，定时任务相对**简单**，且能够接受服务器重启后任务状态丢失，那么 `@Scheduled`是最高效、最便捷的选择。例如，定时刷新本地缓存、清理临时文件等。
    
- **选择 Quartz**：如果你的应用需要**部署多个实例**（集群/分布式），要求任务**不能重复执行**；或者任务需要**持久化**以防重启后丢失；又或者你需要**动态调整**任务执行时间、管理复杂的任务依赖关系。那么，Quartz 是更专业、可靠的选择。例如，每日对账、月度报表生成等关键业务。
    

有一个简洁的口诀可以帮助你记忆：**单机简单用注解，多节点上Quartz。若是任务要可靠，持久化配置不能少**。
